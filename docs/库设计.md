# 库设计

## 1. 结论

库的三层设计思路：

- `dri_xxx`：对外提供“可直接用”的驱动接口，负责组合动作。
- `dri_ll_xxx`：给 `dri_xxx` 提供单外设、低层、稳定的寄存器语义接口。
- `dri_ll.h`：提供最基础的通用寄存器读写原语。

这套分层尤其适合 STM32 裸机项目，因为很多“初始化一个功能”的动作本来就会同时涉及多个模块。例如 GPIO 初始化表面上是配 GPIO，实际上常常还要：

- 开 GPIO 对应时钟
- 根据板级引脚映射选择端口和引脚
- 需要时打开 AFIO 时钟
- 需要时配置重映射

这些动作放在 `dri_xxx` 很合理；真正只操作某一个外设寄存器的动作放到 `dri_ll_xxx` 更清晰。

不过有一个很重要的边界要守住：

- `dri_ll.h` 只放“通用寄存器操作原语”
- GPIO/RCC/SPI/FLASH 的地址、偏移、位定义，仍然分别放在各自的 `dri_ll_xxx.h`

也就是说，`dri_ll.h` 适合放：

```c
dri_ll_write_reg()
dri_ll_read_reg()
dri_ll_set_bits()
dri_ll_clear_bits()
dri_ll_modify_reg()
dri_ll_read_field()
```

但不适合把所有外设的寄存器地址和位定义也都塞进去。否则它会变成一个越来越重的“超级头文件”，后面维护会很累。

## 2. 推荐职责边界

### 2.1 `dri_ll.h`

职责只有一个：提供所有 LL 层都能复用的通用寄存器原语。

推荐规则：

- 只做“地址 + 偏移 + 掩码”的基础操作
- 不感知具体外设
- 不感知板级资源
- 不做业务含义判断
- 不负责时钟开启

你当前仓库里的 [`Libraries/driver/dri_ll.h`](../Libraries/driver/dri_ll.h) 已经基本在这个方向上了，这个思路是对的。

### 2.2 `dri_ll_xxx`

职责是“单外设低层封装”。

以 `dri_ll_gpio` 为例，它应该只关心：

- `GPIOA/GPIOB/...` 的基地址
- `CRL/CRH/IDR/ODR/BSRR/BRR/LCKR` 的偏移
- 某个引脚的配置字段怎么写
- 某个引脚怎么读、怎么置位、怎么复位

它不应该关心：

- 这个引脚在板子上叫不叫 `LED1`
- 这个引脚是不是 `SPI1_SCK`
- 这个引脚对应的时钟该不该自动打开
- 当前项目的 BSP 如何命名

一句话：`dri_ll_gpio` 只面向“GPIO 外设”，不面向“板子上的功能点”。

### 2.3 `dri_xxx`

职责是“对外功能接口”或者“面向板级/模块的接口”。

以 `dri_gpio` 为例，它适合负责：

- 根据 BSP 逻辑名找到对应端口/引脚
- 自动打开 GPIO 时钟
- 需要时自动打开 AFIO 时钟
- 需要时执行 AFIO 重映射
- 把上层配置翻译成 `dri_ll_gpio` 能理解的参数

所以你说的 `dri_init()`、`dri_gpio_init()` 这一类接口，确实就应该放在这一层。

## 3. GPIO 作为示例时，推荐怎么分

### 3.1 推荐目录

建议最终形成类似下面的结构：

```text
Libraries/
├── driver/
│   ├── dri_ll.h
│   ├── dri_ll_gpio.h
│   ├── dri_ll_gpio.c
│   ├── dri_ll_rcc.h
│   ├── dri_ll_rcc.c
│   ├── dri_gpio.h
│   └── dri_gpio.c
└── bsp/
    ├── bsp_gpio.h
    └── bsp_gpio.c
```

如果你暂时不想新建 `bsp/` 目录，也可以先把板级映射单独放在一个文件里，但建议尽快把“板级映射”和“驱动实现”分开。

### 3.2 第 1 层：`dri_ll.h` 应该怎么实现

这一层继续保持极小、极稳定即可。

推荐保留的内容：

- `dri_ll_write_reg()`
- `dri_ll_read_reg()`
- `dri_ll_set_bits()`
- `dri_ll_clear_bits()`
- `dri_ll_modify_reg()`
- `dri_ll_read_field()`
- 必要时补一个 `dri_ll_write_field()`

推荐不放的内容：

- GPIO/RCC/SPI 的具体地址定义
- 某个外设专属的位枚举
- 某个板子的引脚映射

这一层的目标不是“功能丰富”，而是“任何 LL 外设都能放心复用”。

### 3.3 第 2 层：`dri_ll_gpio` 应该怎么实现

#### 3.3.1 这一层的职责

`dri_ll_gpio` 只做 GPIO 外设本身的事情：

- 计算 `GPIOx` 基地址
- 选择 `CRL` 还是 `CRH`
- 计算某个 pin 的 4bit 配置字段位置
- 写 `BSRR/BRR`
- 读 `IDR/ODR`

这一层不负责开时钟。

这点很重要。因为“开时钟”是和系统策略、板级资源绑定的，不是 GPIO 外设本身的寄存器语义。

#### 3.3.2 推荐的 LL 接口

如果你想保持 STM32F1 特征明显、实现简单，那么 `dri_ll_gpio` 可以直接暴露 F1 的 `MODE/CNF` 语义：

```c
typedef struct
{
    dri_ll_gpio_port_t  port;
    dri_ll_gpio_pin_t   pin;
    dri_ll_gpio_mode_t  mode;
    dri_ll_gpio_cnf_t   cnf;
    dri_ll_gpio_level_t init_level;
} dri_ll_gpio_init_t;

void dri_ll_gpio_init(const dri_ll_gpio_init_t* config);
void dri_ll_gpio_set_pin(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin);
void dri_ll_gpio_reset_pin(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin);
void dri_ll_gpio_write_pin(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin, dri_ll_gpio_level_t level);
dri_ll_gpio_level_t dri_ll_gpio_read_input(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin);
dri_ll_gpio_level_t dri_ll_gpio_read_output(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin);
void dri_ll_gpio_toggle_pin(dri_ll_gpio_port_t port, dri_ll_gpio_pin_t pin);
```

这样做的好处是：

- 非常贴近 F1 的寄存器模型
- 实现简单
- 不会把 `MODE/CNF` 的转换逻辑散落到外层

如果你后面希望跨系列复用 GPIO 接口，再考虑把 `mode/cnf` 换成更抽象的“方向/复用/上下拉/输出类型/速度”模型。

#### 3.3.3 `dri_ll_gpio.c` 的内部实现建议

推荐拆出几个内部函数：

```c
static uptr dri_ll_gpio_get_base(dri_ll_gpio_port_t port);
static uptr dri_ll_gpio_get_cfg_offset(dri_ll_gpio_pin_t pin);
static u32  dri_ll_gpio_get_cfg_shift(dri_ll_gpio_pin_t pin);
static u32  dri_ll_gpio_get_pin_mask(dri_ll_gpio_pin_t pin);
```

其中核心逻辑是：

- `pin < 8` 时操作 `CRL`
- `pin >= 8` 时操作 `CRH`
- 每个 pin 对应 4bit 配置字段
- 字段位移为 `(pin % 8) * 4`

初始化函数的核心写法可以是：

```c
offset = (pin < 8) ? DRI_LL_GPIO_CRL_OFFSET : DRI_LL_GPIO_CRH_OFFSET;
shift  = (pin & 0x7U) * 4U;
clear_mask = 0xFUL << shift;
set_mask   = ((((u32)config->cnf << 2U) | (u32)config->mode) & 0x0FU) << shift;

dri_ll_modify_reg(base, offset, clear_mask, set_mask);
```

#### 3.3.4 F1 上 GPIO 初始化要注意的细节

STM32F1 的 GPIO 和 F4/F0 那种 `MODER/OTYPER/PUPDR` 不一样，要特别注意：

- 输入模式时，`MODE=00`
- 输出速度其实编码在 `MODE` 里
- `CNF` 在输入模式和输出模式下含义不同
- 输入上拉/下拉不是独立寄存器位，而是 `CNF=10` 配合 `ODR` 位决定

所以 `dri_ll_gpio_init()` 推荐按下面的顺序处理：

1. 先算出 `base/offset/shift`
2. 如果是输入上拉/下拉模式，先根据 `init_level` 设置 `ODR`
3. 再写 `CRL/CRH`
4. 如果是输出模式，最后再根据 `init_level` 设置默认输出电平

这里 `init_level` 在两种场景下都能复用：

- 对输出模式：表示初始输出电平
- 对输入上拉/下拉模式：`HIGH` 表示上拉，`LOW` 表示下拉

#### 3.3.5 推荐优先使用 `BSRR/BRR`

对单 pin 置位/复位时，建议优先这样做：

- 置位：写 `BSRR`
- 复位：写 `BRR`

这样比直接改 `ODR` 更适合单 pin 操作，也能避免很多读改写问题。

`toggle` 比较特殊，因为 F1 没有单独的“翻转寄存器”。所以 `toggle` 本质上还是：

1. 读 `ODR`
2. 判断当前位
3. 再写 `BSRR` 或 `BRR`

如果你后面担心并发访问，可以把 `toggle` 视为“非原子操作”，或者在很短的临界区里做。

#### 3.3.6 当前 `dri_ll_gpio.h` 的一个优化建议

你现在的 [`Libraries/driver/dri_ll_gpio.h`](../Libraries/driver/dri_ll_gpio.h) 里把 `CRL/CRH` 的每一位几乎都逐项展开了。这个做法不是错，但长期维护性不太高。

对 GPIO 这种规律很强的外设，更推荐：

- 保留基地址、寄存器偏移、必要的枚举类型
- 把“每个 pin 对应哪几位”的计算放进 `.c` 里的内部函数

原因很简单：

- 代码量更小
- 不容易写错或复制错
- 后面读代码时更容易把注意力放在“算法”和“职责”上

### 3.4 第 3 层：`dri_gpio` 应该怎么实现

#### 3.4.1 这一层的职责

这一层是对外接口，面向“板子上的资源名称”或者“模块初始化动作”。

它要把多个低层动作串起来。

例如 `LED1` 初始化，可能同时涉及：

- `RCC_APB2ENR` 打开 `GPIOC`
- `GPIOC.13` 设置成推挽输出
- 根据板子极性决定默认输出电平

这已经不是一个“纯 GPIO 外设寄存器动作”了，所以就应该放在 `dri_gpio`。

#### 3.4.2 推荐增加 BSP 映射层

既然你已经明确提到了“根据 BSP 中的引脚映射配置对应引脚”，那建议把这部分显式建成一层数据描述，而不是直接写死在 `dri_gpio.c` 里。

例如：

```c
typedef enum
{
    BSP_GPIO_LED1,
    BSP_GPIO_KEY1,
    BSP_GPIO_SPI1_SCK,
    BSP_GPIO_SPI1_MISO,
    BSP_GPIO_SPI1_MOSI,
} bsp_gpio_id_t;

typedef struct
{
    dri_ll_gpio_port_t     port;
    dri_ll_gpio_pin_t      pin;
    dri_ll_rcc_apb2enr_bits gpio_clk_mask;
    dri_ll_gpio_mode_t     mode;
    dri_ll_gpio_cnf_t      cnf;
    dri_ll_gpio_level_t    init_level;
    bool                   need_afio_clk;
} bsp_gpio_desc_t;

const bsp_gpio_desc_t* bsp_gpio_get_desc(bsp_gpio_id_t id);
```

这样以后你换板子时：

- 改 BSP 映射表
- `dri_gpio` 和 `dri_ll_gpio` 都不用大改

#### 3.4.3 推荐的对外接口

如果 `dri_gpio` 是给应用层直接用的，推荐用“逻辑资源名”而不是“裸 port/pin”：

```c
typedef enum
{
    DRI_GPIO_LED1,
    DRI_GPIO_KEY1,
} dri_gpio_id_t;

typedef enum
{
    DRI_GPIO_OK = 0,
    DRI_GPIO_ERROR_PARAM,
    DRI_GPIO_ERROR_UNSUPPORTED,
} dri_gpio_status_t;

dri_gpio_status_t dri_gpio_init(dri_gpio_id_t id);
dri_gpio_status_t dri_gpio_write(dri_gpio_id_t id, dri_ll_gpio_level_t level);
dri_ll_gpio_level_t dri_gpio_read(dri_gpio_id_t id);
dri_gpio_status_t dri_gpio_toggle(dri_gpio_id_t id);
```

如果你还想保留“按 port/pin 直接操作”的能力，也可以同时提供另一组接口，但那一组更像是高级版 `dri_ll_gpio`，不建议作为主要对外入口。

#### 3.4.4 `dri_gpio_init()` 的典型流程

推荐流程如下：

1. 根据 `id` 去 BSP 查找描述信息
2. 打开 GPIO 对应时钟
3. 如果需要复用或重映射，再决定是否打开 AFIO 时钟
4. 组装 `dri_ll_gpio_init_t`
5. 调用 `dri_ll_gpio_init()`
6. 返回状态

伪代码可以写成：

```c
dri_gpio_status_t dri_gpio_init(dri_gpio_id_t id)
{
    const bsp_gpio_desc_t* desc = bsp_gpio_get_desc((bsp_gpio_id_t)id);
    dri_ll_gpio_init_t ll_cfg;

    if (desc == NULL)
    {
        return DRI_GPIO_ERROR_PARAM;
    }

    dri_ll_rcc_apb2_enable(desc->gpio_clk_mask);

    if (desc->need_afio_clk)
    {
        dri_ll_rcc_apb2_enable(RCC_APB2_AFIO);
    }

    ll_cfg.port       = desc->port;
    ll_cfg.pin        = desc->pin;
    ll_cfg.mode       = desc->mode;
    ll_cfg.cnf        = desc->cnf;
    ll_cfg.init_level = desc->init_level;

    dri_ll_gpio_init(&ll_cfg);
    return DRI_GPIO_OK;
}
```

这正好体现你的设计核心：

- `dri_gpio_init()` 可以操作多个外设寄存器
- `dri_ll_gpio_init()` 只关心 GPIO
- 最底层真正的寄存器读改写都统一经过 `dri_ll.h`

## 4. 三层之间的调用关系

以 `LED1` 为例，推荐的调用链是：

```text
应用层
  -> dri_gpio_init(DRI_GPIO_LED1)
      -> bsp_gpio_get_desc(DRI_GPIO_LED1)
      -> dri_ll_rcc_apb2_enable(RCC_APB2_GPIOC)
      -> dri_ll_gpio_init(...)
          -> dri_ll_modify_reg(...)
          -> dri_ll_write_reg(...)
```

再例如写 LED：

```text
应用层
  -> dri_gpio_write(DRI_GPIO_LED1, GPIO_LEVEL_HIGH)
      -> bsp_gpio_get_desc(...)
      -> dri_ll_gpio_write_pin(GPIO_PORT_C, GPIO_PIN_13, GPIO_LEVEL_HIGH)
          -> dri_ll_write_reg(GPIOC_BASE, BSRR/BRR, ...)
```

这样的调用关系有三个优点：

- 上层接口好用
- 中层接口稳定
- 最底层寄存器原语统一

## 5. 这套设计在你当前项目里的落地建议

结合你仓库里现在的情况，我建议这样推进：

### 5.1 保持 `dri_ll.h` 的定位不变

你当前的 [`Libraries/driver/dri_ll.h`](../Libraries/driver/dri_ll.h) 已经是“通用寄存器原语层”了，继续保持这个方向即可，不要把它扩成外设大杂烩。

### 5.2 `dri_ll_rcc`、`dri_ll_flash` 的方向是对的

你现在的 [`Libraries/driver/dri_ll_rcc.c`](../Libraries/driver/dri_ll_rcc.c) 和 [`Libraries/driver/dri_ll_flash.c`](../Libraries/driver/dri_ll_flash.c) 已经在做“单外设 LL 封装”，这和上面的设计是一致的。

### 5.3 `dri_ll_gpio` 建议从“位定义堆积”转向“接口 + 内部计算”

GPIO 最适合做成：

- 头文件里保留必要类型和函数声明
- `.c` 里用内部函数计算 `CRL/CRH` 的位置和掩码

不建议继续把每个 pin 的 `MODE/CNF` 位全部枚举到很细。

### 5.4 真正的“自动开时钟 + 板级映射”放到 `dri_gpio`

这件事不要放到 `dri_ll_gpio`。

因为一旦放进去，`dri_ll_gpio` 就会开始知道：

- GPIO 之外的 RCC
- 板级资源命名
- 某个工程的固定引脚分配

这样它很快就不再是 LL 层了。

## 6. 最后给你的结论

你的想法是可行的，而且方向是对的。更准确地说：

- `dri_xxx` 负责“对外功能接口”和“跨外设组合动作”
- `dri_ll_xxx` 负责“单外设低层语义”
- `dri_ll.h` 负责“通用寄存器原语”

如果以 GPIO 为例，最推荐的落地方式是：

- `dri_gpio` 读取 BSP 映射并自动开时钟
- `dri_ll_gpio` 只负责 `CRL/CRH/IDR/ODR/BSRR/BRR`
- `dri_ll.h` 只负责寄存器读写、位操作、字段操作

这样分完之后，代码会有两个很明显的好处：

- 对外接口更好用，应用层不用关心寄存器细节
- 低层更稳定，后面写 SPI/UART/TIM 时可以复用同一套思路
